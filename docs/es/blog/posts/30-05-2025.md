# Explotando Vulnerabilidades Comunes en JWT: Laboratorio Práctico

Los JSON Web Tokens (JWT) se utilizan ampliamente para autenticación y autorización en aplicaciones web modernas. Sin embargo, las malas configuraciones y fallos de implementación pueden introducir vulnerabilidades críticas de seguridad. Esta publicación explora un entorno de laboratorio práctico que demuestra cuatro vulnerabilidades comunes en JWT y proporciona instrucciones paso a paso para su explotación.

## ¿Qué son las Json Web Tokens?

Son un estándar abierto para la creación de tokens que representa información sobre un usuario (identidad y roles), y principalmente son utilizados en aplicaciones modernas como un mecanismo de autenticación sin estado.

Lo conforman 3 partes:

- **Cabecera**: Indica el tipo de token y el algoritmo de firma usado.
- **Payload**: Contiene los datos del usuario y metadatos adicionales (rol, nombre, expiración, etc).
- **Firma**: Firma criptográfica que valida la integridad del token y asegura que no ha sido modificado.

Cada token está codificado en Base64, lo que permite que sea transmitido por _HTTP_ con facilidad.

## Descripción del Laboratorio

El laboratorio consiste en un backend Node.js con endpoints para registro, inicio de sesión, perfil y acceso de administrador. El backend es intencionalmente vulnerable a los siguientes ataques:

1. **Bypass alg:none**
2. **Inyección en kid**
3. **Confusión de Algoritmo**
4. **Manipulación del Payload**

Obtendrás una bandera por cada vulnerabilidad explotada, la cual se revela tras una explotación exitosa.

## 1. Bypass alg:none

### Descripción

El backend acepta tokens con el encabezado `alg: "none"`, permitiendo a los atacantes forjar tokens sin una firma válida.

### Proceso de explotación

1. **Obtén un JWT válido** iniciando sesión como usuario regular.
2. **Decodifica el token** y cambia el algoritmo en el encabezado a `"none"`.
3. **Elimina la firma** (la tercera parte del JWT).
4. **Establece el claim `isAdmin` en `true`** en el payload.
5. **Envía el token modificado** al endpoint `/profile` o `/admin`.

Identificamos que el token de sesión se pasa vía cookie. Luego usamos un [editor de JWT](https://token.dev/){:target="\_blank"} para decodificar y modificar el token:

Para explotar la vulnerabilidad del algoritmo, modificamos el encabezado de `"alg": "HS256"` a `"alg": "none"` y escalamos privilegios estableciendo `"isAdmin": true`:

```bash
{
"alg": "none",
"typ": "JWT",
"user": "jose",
"isAdmin": true,
"iat": 1748629270
}
```

Copiamos el token modificado en Base64 en Burp Suite y reenviamos la petición. Esto otorga acceso al endpoint `/admin` en lugar de `/profile`, demostrando la escalada de privilegios:

  <figcaption>Acceso de administrador logrado mediante token modificado</figcaption>
</figure>

#### Mitigación

Para poder mitigar este tipo de ataques es una buena práctica **validar siempre** el algoritmo esperado antes de procesar el token.

```js
const decoded = jwt.verify(token, process.env.TOKEN_SECRET, {
  algorithms: ["HS256", "HS384", "HS512"],
});
```

Con esto hacemos que solamente se procesen tokens cuyos algoritmos son los indicados, haciendo que sea imposible procesar un token con algoritmo **none**.

## 2. Inyección en kid

### Descripción

El backend verifica un valor específico de `kid` (Key ID) en el encabezado del JWT, simulando una vulnerabilidad de inyección tipo SQL.

### Proceso de explotación

1. **Crea un JWT con un encabezado `kid` malicioso**:

```bash
{
"alg": "HS256",
"typ": "JWT",
"kid": "<payload_malicioso>"
}
```

2. Además indicamos en el campo `isAdmin` que sea verdadero para poder acceder al endpoint de Admin Panel.
3. **Firma el token** con el secreto correcto.
4. **Envía el token** al endpoint `/profile`, que nos redireccionará al `/AdminPanel`.

### Explotación mediante path traversal

Primero, como en el caso anterior, iniciamos sesión y obtenemos el token correspondiente a nuestro usuario. Luego, haciendo uso de un [editor de JWT](https://token.dev/){:target="\_blank"} decodificamos y modificamos el `kid` del token:

Ahora, iniciamos sesión haciendo uso del proxy Burp Suite insertando el token modificado y como resultado tenemos:

  <figcaption>Acceso al admin panel via Kid Injection con path traversal</figcaption>

#### Mitigación

Para poder evitar este tipo de ataque, podríamos:

- **Validar estrictamente el valor de `kid`**: Haciendo uso de whitelists para permitir únicamente valores permitidos.

```js
const validKids = new Set(["key1", "key2"]);
if (!validKids.has(decoded.header.kid)) {
  throw new Error("Invalid key ID");
}
```

- **Sanitizar la entrada**: No permitir caracteres especiales, como podría ser las rutas relativas (`../`), ni rutas absolutas.
  ```js
  function sanitizeKid(kid) {
    return kid.replace(/[\.\/\\'"\s;]/g, "");
  }
  ```

### Explotación mediante inyección SQL

Suponemos que el backend de la aplicación web verifica la firma del token mediante la siguiente consulta SQL:

```sql
SELECT key FROM keys WHERE key_id = '[kid]';
```

Un atacante podría manipular el token de la siguiente manera:

<figure markdown="span">
    ![Modificación token kid SQLi](../../assets/posts/2025/05/30/JWT7.png)
  <figcaption>Modificación del kid para SQLi</figcaption>
</figure>

Al enviar el token modificado, la consulta SQL que se realiza en el backend sería la siguiente:

```sql
SELECT key FROM keys WHERE key_id = '1' UNION SELECT 'clave'--';
```

Esto hace que el backend devuelva la cadena 'clave' como la firma. Si esto sucede, el atacante podría firmar el token JWT con esa clave y forzar al backend a aceptar un token "malicioso" como válido.

#### Mitigación

Para prevenir la explotación mediante inyección SQL en el parámetro kid del JWT, implementa las siguientes medidas:

- **Preparar o parametrizar las consultas**: Utiliza consultas preparadas que separan los datos del código SQL, evitando así la ejecución de comandos arbitrarios.

```js
const query = "SELECT key FROM keys WHERE key_id = ?";
db.query(query, [kid], (err, results) => {
  //..
});
```

- **Validación y saneamiento del valor del `kid`**: Podemos indicar que el kid solamente contenga caracteres permitidos y un formato esperado.

```js
const validKids = new Set(["key1", "key2"]);
if (!validKids.has(kid)) {
  throw new Error("Invalid key ID");
}
```

## 3. Confusión de Algoritmo

### Descripción

El backend permite que el token indique el algoritmo de firma y utiliza la clave pública como secreto para verificar tokens firmados con algoritmo simétrico (HS256). Esto permite a un atacante firmar tokens con la clave pública y que el servidor los acepte como válidos.

### Proceso de explotación

1. Obtener la clave pública del servidor (por ejemplo, desde `/jwks.json`).

2. Crear un token con algoritmo HS256 y el payload deseado.

3. Firmar el token usando la clave pública como secreto.

4. Enviar el token al endpoint protegido.

### Explotación

#### Mitigación

- Validar explícitamente el algoritmo esperado:

```js
jwt.verify(token, publicKey, { algorithms: ["RS256"] });
```

- No mezclar claves públicas y secretos simétricos.

## 4. Manipulación del Payload

### Descripción

El backend concede privilegios de administrador si el campo isAdmin es true en el payload, sin validar adecuadamente la firma o el origen del token

### Pasos de Explotación

1. Obtener un token válido (por ejemplo, iniciando sesión como usuario normal).

2. Modificar el payload para establecer isAdmin: true.

3. Re-firmar el token con el secreto correcto (si es necesario, o aprovechar otras vulnerabilidades para evitar la firma).

4. Enviar el token al endpoint protegido.

### Explotación


#### Mitigación
- Validar siempre la firma del token.

- No confiar únicamente en el payload para la autorización.

- Validar claims críticos como `iss`, `aud`, `exp`, etc.
